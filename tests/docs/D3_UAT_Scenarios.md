## สถานการณ์ที่ 1: เจ้าหน้าที่บันทึกมิเตอร์
- วัตถุประสงค์: เจ้าหน้าที่บันทึกค่ามิเตอร์ได้ถูกต้อง
- ขั้นตอน:
1. Login ด้วยบัญชี staff
2. เปิด endpoint POST /api/meter-readings
3. กรอกค่าก่อนหน้า/ปัจจุบันของน้ำและไฟ
4. ส่งคำขอ
- ผลลัพธ์ที่คาดหวัง:
1. ระบบตอบ 201
2. ได้ readingId ใหม่
3. ข้อมูลนำไปใช้สร้างบิลได้
- ผลการทดสอบ: PASS

## สถานการณ์ที่ 2: ฝ่ายการเงินออกใบแจ้งหนี้
- วัตถุประสงค์: ฝ่ายการเงินออกใบแจ้งหนี้จากข้อมูลมิเตอร์ล่าสุด
- ขั้นตอน:
1. Login ด้วยบัญชี finance
2. เรียก POST /api/invoices/batch พร้อมระบุ `month`
3. ตรวจผลรวมค่าเช่า ค่าน้ำ ค่าไฟ ของ invoice ที่สร้างขึ้น
- ผลลัพธ์ที่คาดหวัง:
1. ระบบตอบ 201
2. ระบบสร้าง invoices ได้มากกว่า 0 รายการ
3. ยอดรวมคำนวณถูกต้องและสถานะเริ่มต้นเป็น `pending`
- ผลการทดสอบ: PASS

## สถานการณ์ที่ 3: นักศึกษาแจ้งชำระเงินและฝ่ายการเงินตรวจสอบรายการ
- วัตถุประสงค์: ทดสอบ workflow ตั้งแต่การแนบสลิปจนถึงการอนุมัติการชำระเงิน
- ขั้นตอน:
1. Login ด้วยบัญชี student และเรียก POST /api/payments/submit พร้อม `invoiceId` และ `slipUrl`
2. Login ด้วยบัญชี finance แล้วเรียก GET /api/payments/pending
3. เรียก POST /api/payments/:paymentId/decision ด้วยสถานะ `approved`
- ผลลัพธ์ที่คาดหวัง:
1. ระบบบันทึกรายการชำระเงินได้และตั้งสถานะเริ่มต้นเป็น `pending`
2. ฝ่ายการเงินเห็นรายการใน pending queue
3. เมื่ออนุมัติแล้ว ใบแจ้งหนี้ที่เกี่ยวข้องเปลี่ยนเป็น `paid`
- ผลการทดสอบ: PASS

## สถานการณ์ที่ 4: ตรวจสอบสิทธิ์ตามบทบาท
- วัตถุประสงค์: ตรวจว่าบทบาทที่ไม่เกี่ยวข้องถูกปฏิเสธ
- ขั้นตอน:
1. Login ด้วยบัญชี staff
2. เรียก GET /api/payments/pending
- ผลลัพธ์ที่คาดหวัง:
1. ระบบตอบ 403 FORBIDDEN
- ผลการทดสอบ: PASS

## สถานการณ์ที่ 5: ตรวจสอบการจัดการคำขอที่ไม่ถูกต้อง
- วัตถุประสงค์: ตรวจ error response เมื่อ payload ไม่ครบ
- ขั้นตอน:
1. Login ด้วยบัญชี student แล้วเรียก POST /api/payments/submit โดยไม่ส่ง `slipUrl`
- ผลลัพธ์ที่คาดหวัง:
1. ระบบตอบ 400 BAD REQUEST
2. ข้อความ error ระบุว่าต้องส่ง `slip image`
- ผลการทดสอบ: PASS
