## สถานการณ์ที่ 1: เจ้าหน้าที่บันทึกมิเตอร์
- วัตถุประสงค์: เจ้าหน้าที่บันทึกค่ามิเตอร์ได้ถูกต้อง
- ขั้นตอน:
1. Login ด้วยบัญชี staff
2. เปิด endpoint POST /api/meter-readings
3. กรอกค่าก่อนหน้า/ปัจจุบันของน้ำและไฟ
4. ส่งคำขอ
- ผลลัพธ์ที่คาดหวัง:
1. ระบบตอบ 201
2. ได้ readingId ใหม่
3. ข้อมูลนำไปใช้สร้างบิลได้
- ผลการทดสอบ: PASS

## สถานการณ์ที่ 2: ฝ่ายการเงินออกใบแจ้งหนี้
- วัตถุประสงค์: ฝ่ายการเงินออกใบแจ้งหนี้จากข้อมูลมิเตอร์ล่าสุด
- ขั้นตอน:
1. Login ด้วยบัญชี finance
2. เรียก POST /api/invoices พร้อม roomId และ dueDate
3. ตรวจผลรวมค่าเช่า ค่าน้ำ ค่าไฟ
- ผลลัพธ์ที่คาดหวัง:
1. ระบบตอบ 201
2. grandTotal คำนวณถูกต้อง
3. สถานะเริ่มต้นเป็น pending
- ผลการทดสอบ: PASS

## สถานการณ์ที่ 3: ฝ่ายการเงินบันทึกการชำระเงิน
- วัตถุประสงค์: บันทึกการชำระเงินและตรวจสอบสถานะ
- ขั้นตอน:
1. Login ด้วยบัญชี finance
2. เรียก POST /api/payments
3. เรียก PATCH /api/payments/:paymentId/verify ด้วย verified
- ผลลัพธ์ที่คาดหวัง:
1. บันทึกการชำระเงินได้
2. ตรวจสอบสถานะสำเร็จ
3. ใบแจ้งหนี้เปลี่ยนเป็น paid
- ผลการทดสอบ: PASS

## สถานการณ์ที่ 4: ตรวจสอบสิทธิ์ตามบทบาท
- วัตถุประสงค์: ตรวจว่าบทบาทที่ไม่เกี่ยวข้องถูกปฏิเสธ
- ขั้นตอน:
1. Login ด้วยบัญชี staff
2. เรียก GET /api/payments
- ผลลัพธ์ที่คาดหวัง:
1. ระบบตอบ 403 FORBIDDEN
- ผลการทดสอบ: PASS

## สถานการณ์ที่ 5: ตรวจสอบการจัดการคำขอที่ไม่ถูกต้อง
- วัตถุประสงค์: ตรวจ error response เมื่อ payload ไม่ครบ
- ขั้นตอน:
1. เรียก POST /api/rooms โดยไม่ส่ง typeId
- ผลลัพธ์ที่คาดหวัง:
1. ระบบตอบ 400 VALIDATION_ERROR
2. ข้อความ error ชัดเจน
- ผลการทดสอบ: PASS
